2. Project Vision & Goals
2.1 Vision Statement
Whisper Crystals is a narrative-driven 2D side-scrolling space pirate game inspired by the Spelljammer D&D setting. The player commands Aristotle, a self-made cat pirate captain who controls the universe's only known source of starship fuel — Whisper Crystals. The game spans a multiverse of colliding factions from fantasy, sci-fi, and myth, all competing for control of this resource.
AGENT INSTRUCTION: Use this vision statement verbatim as the opening paragraph in PRD-001 Section 1 (Product Overview).
2.2 Core Goals
	•	Deliver a playable 2D prototype in Python/Pygame demonstrating core game loop, narrative choice, and faction interaction.
	•	Establish a codebase architecture that can migrate cleanly to Godot or Unity in a future phase.
	•	Create a rich, character-driven narrative with meaningful player choices across 4 story arcs.
	•	Build a commodity economy system around Whisper Crystals that drives all political and conflict mechanics.
	•	Support future web-based collaboration tooling to allow co-development and shared game state visibility.

AGENT INSTRUCTION: Translate each core goal into a numbered functional requirement with MoSCoW priority in PRD-001. Prototype delivery goals are MUST HAVE. Migration and collaboration tooling are SHOULD HAVE or COULD HAVE.
2.3 Out of Scope (Phase 1)
	•	3D graphics or 3D game engine (Phase 2+)
	•	Multiplayer or online play
	•	Mobile platform builds
	•	Full voice acting or orchestral soundtrack
	•	Unity or Godot migration (evaluation only in Phase 1)

3. World & Narrative Context
3.1 Setting
A vast multiverse of interconnected realms — fantasy kingdoms, sci-fi civilisations, alien worlds, mythological planes — all linked by interstellar space. Ships travel between realms using Whisper Crystals as fuel. No crystals, no travel. The faction that controls crystals controls the multiverse.
Each realm produces its own faction type with entirely distinct political systems, combat abilities, and cultural values. These factions do not merely coexist — every single one is actively competing to become the dominant power across all realms. The goal for every faction is conquest and total control of the multiverse. Aristotle sits at the centre of this because whoever controls the crystals controls who can even participate in that race.
	•	Realm of the Feline Courts — produces Felid Corsairs and Lions. Politics: meritocracy vs noble birthright. Abilities: stealth, speed, cunning diplomacy.
	•	Realm of the Canine Order — produces Canis League and Wolves. Politics: military hierarchy and pack loyalty. Abilities: organised warfare, siege tactics, superior logistics.
	•	Fairy Realms — information, illusion, and trade. Politics: fluid mercantile allegiances. Abilities: magical espionage, enchanted goods, intelligence networks.
	•	Knight Kingdoms — chivalric order and law. Politics: feudal hierarchy imposing universal order. Abilities: heavy armour, disciplined formations, siege engineering.
	•	Goblin Warrens — scavengers and engineers. Politics: anarchic guild cartels. Abilities: rapid ship modification, black-market crystal trade, sabotage.
	•	Alien Space Races — advanced deep-space civilisations. Politics: varies wildly by race. Abilities: superior technology, unknown weapons, deeply unpredictable behaviour.

AGENT INSTRUCTION: In PRD-003, create a Faction Unique Abilities requirement. Each faction must have at least 3 distinct mechanical abilities differentiating their ships, crew, and encounter options. These must affect gameplay outcomes, not just aesthetics. In TRD-003, model faction_abilities[] as a typed ability registry per faction entity.
3.2 The Commodity — Whisper Crystals
	•	Sole known fuel source for inter-realm space travel (equivalent to dilithium crystals in Star Trek or spice in Dune).
	•	Exclusively produced and refined by the Felid Corsairs (cat faction), discovered by Aristotle.
	•	Every faction is dependent on cat supply — this is the central economic and political tension.
	•	Crystals have a subtle mystical quality: they whisper. The lore behind this is deliberately mysterious.

AGENT INSTRUCTION: In PRD-002, create a Commodity System feature requirement covering: crystal inventory management, supply/demand mechanics, trade negotiation events, and supply route attack/defence scenarios. In TRD-003, model Whisper Crystals as a core game resource with quantity, quality, and location attributes.

	•	Canis League — dogs. Hierarchical military structure, heavy ships, crystal-dependent. Ideology: order through loyalty.

	•	Goblins — black-market engineers and scavengers. Will sell modified ship parts, stolen crystals, or sabotage services to the highest bidder.
	•	Alien Space Races — wildcard factions from deep space with their own agendas and technologies that others do not yet understand.

5. Core Game Mechanics
5.1 Game Loop
The core loop is: Navigate space → Encounter event (combat / trade / diplomacy) → Make decision → Outcome affects resources, faction relationships, and story state → Progress arc → Repeat.
AGENT INSTRUCTION: In PRD-001, define this as the Core Game Loop requirement with entry/exit states for each loop phase. In TRD-001, map each phase to a Python module or class. The game loop must be the central orchestrator.
5.2 Ship System
	•	Cat ships: Fast, low armour, high manoeuvrability. Asymmetric design.
	•	Dog ships: Bulkier, more guns, stronger shields. Naval structure.
	•	Ship stats: Speed, Armour, Firepower, Crystal Capacity, Crew Capacity.
	•	Ships can be damaged, upgraded, or captured during encounters.

AGENT INSTRUCTION: In PRD-001, create Ship System requirements covering: ship stats, damage model, upgrade paths, and fleet management. In TRD-003, model ships as entities with stat attributes and an equipment/upgrade slot system.
5.3 Encounter System
	•	Combat encounters: Side-scrolling space combat. Player fires, dodges, boards enemy ships.
	•	Trade encounters: Negotiation UI. Offer/counter-offer on crystal supply. Faction relationship affects pricing.
	•	Diplomatic encounters: Dialogue trees. Choice-driven. Outcomes affect faction standing and story flags.
	•	Exploration encounters: Discover new realms, hidden crystal deposits, lore artefacts.

AGENT INSTRUCTION: In PRD-001, define four Encounter Type requirements (Combat, Trade, Diplomatic, Exploration), each with input triggers, player interaction model, success/failure outcomes, and effects on game state. In TRD-002, specify the encounter engine as an event-driven system with an encounter_type dispatcher.
5.4 Narrative Choice System
	•	Player decisions at key story beats alter faction relationships, resource access, and available arcs.
	•	Choices are not binary good/evil — they are strategic tradeoffs (short-term vs long-term gain).
	•	Arc 4 endings (Hold / Share / Destroy) are determined by cumulative choice history, not a single decision.

AGENT INSTRUCTION: In PRD-002, create a Narrative Choice System requirement. Specify: choice presentation (dialogue UI), consequence tracking (flags and scores), and ending calculation logic (weighted choice history). In TRD-003, model choice history as a persistent player_decisions[] array with outcome_weights.

6. Migration Strategy (Non-Negotiable)
CRITICAL REQUIREMENT: The Phase 1 Python/Pygame prototype MUST be architected for clean migration to a commercial game engine (Godot 4 or Unity) in a later phase. This is not optional. Every technical decision in Phase 1 must be made with migration in mind. Any code that cannot be migrated must be explicitly documented as throwaway.
6.1 Migration Mandate
The prototype is a proof of concept and narrative vehicle — not the final product. The architecture must ensure that when the decision is made to migrate to Godot 4 or Unity, the core game logic, data models, story content, and asset pipeline transfer with minimal rework. The engine is replaceable. The game is not.
AGENT INSTRUCTION: In TRD-001, elevate Migration Readiness to a first-class architectural constraint. Every module specification must include a 'Migration Notes' field stating whether it is: PORTABLE (pure logic, transfers as-is), REWRITE REQUIRED (engine-specific, must be rebuilt), or ABSTRACTED (wrapped behind an interface to isolate engine dependency).
6.2 Migration-Ready Design Principles
	•	Engine Abstraction Layer (EAL): All Pygame-specific code (rendering, input, audio, windowing) must sit behind an abstraction interface. Core game logic must never call Pygame directly.
	•	Data-Driven Content: All story arcs, dialogue trees, encounter tables, faction data, and character stats must be defined in JSON/YAML files — not hardcoded. These files migrate without changes.
	•	Pure Python Game Logic: The game state machine, combat calculations, economy model, and narrative engine must be written in pure Python with zero engine dependencies. These become the migration payload.
	•	Asset Independence: All art assets (sprites, backgrounds, UI) must be stored in a structured /assets directory with an asset manifest. The manifest format must be engine-agnostic.
	•	No Engine-Specific Patterns: Avoid Pygame-specific design patterns (e.g. Sprite groups as primary architecture). Use standard OOP patterns that map cleanly to Godot Nodes or Unity GameObjects.

AGENT INSTRUCTION: In TRD-001 Section 3, expand Architecture Principles to include the Engine Abstraction Layer as a required component. Provide a Python interface definition (abstract base class) for the rendering, input, and audio subsystems that Pygame implements in Phase 1 and Godot/Unity implements in Phase 2.

6.4 Migration Trigger Criteria
Migration from Python/Pygame to a commercial engine will be triggered when one or more of the following conditions are met:
	•	The prototype successfully demonstrates all four story arcs end-to-end.
	•	Performance limitations of Pygame prevent delivery of required visual quality or frame rate.
	•	A collaborator or publisher requires a specific engine as a delivery condition.
	•	The decision is made to target console or mobile platforms.

AGENT INSTRUCTION: Include these migration triggers in TRD-001 as the Migration Decision Framework. Add a recommendation: evaluate Godot 4 first (open source, Python-adjacent GDScript, lighter overhead) before Unity (stronger commercial ecosystem, heavier tooling).

7.2 Architecture Principles
	•	Separation of concerns: Game logic, rendering, and data must be cleanly separated from day one.
	•	Migration-ready: All game logic must be engine-agnostic so it can transfer to Godot/Unity. No Pygame-specific logic embedded in core game systems.
	•	State machine driven: Game state (menus, navigation, encounter, cutscene, end) managed via explicit state machine, not ad-hoc flags.
	•	Data-driven content: Story arcs, dialogue, encounter tables loaded from JSON/YAML, not hardcoded.

AGENT INSTRUCTION: In TRD-001, translate each architecture principle into a technical constraint with rationale and implementation guidance. Include a recommended module/package structure diagram (described in text if diagramming is not available).
7.3 Development Philosophy — Vibe Coding
This project is built using a rapid prototyping philosophy sometimes called vibe coding: build fast, see results quickly, iterate on what works. This is a personal project and creative endeavour — not an enterprise delivery. The TRDs must reflect this.
	•	Speed over perfection in Phase 1. Get the game loop running and playable before engineering elegantly.
	•	Iterative builds: each coding session should produce something runnable. No long dark periods of build-up before a first result.
	•	Placeholder-first: placeholder sprites, placeholder audio, placeholder UI — get the logic working first, polish later.
	•	Single developer primary: architecture must be comprehensible to one person. Avoid over-engineering.
	•	AI-assisted development: code will be generated with Claude assistance. TRD code examples should be prompt-ready — clear, well-commented, easy for an AI to extend.

AGENT INSTRUCTION: In TRD-002, add a Development Approach section reflecting the vibe coding philosophy. Recommend a build sequence that delivers a visible, playable result within the first session (e.g. player ship on screen, moving, firing). Structure the TRD so each feature can be built and tested independently before the next is added. Code examples must be clean, commented, and AI-extension-friendly.
7.4 Collaboration Application
A web-based collaboration application is planned to allow a co-creator (friend and creative collaborator) to view and engage with the game's development in real time. This is not a multiplayer game feature — it is a development collaboration tool.
	•	Use case: Developer builds the game locally. Collaborator accesses a web app to see current game state, story progress, faction data, and character status — without needing to run the game themselves.
	•	The collaborator contributed significant creative ideas to this project. The app gives them visibility and engagement in the development process.
	•	Minimum viable collaboration features: view current story arc status, view faction relationship matrix, view character states, view Whisper Crystal economy snapshot.
	•	Phase 3 stretch: collaborator can submit narrative suggestions or faction event ideas via the web app that feed into the development pipeline.

AGENT INSTRUCTION: In PRD-001, add a Collaboration Web Application requirement (COULD HAVE / Phase 3). Note this is a companion tool for co-creator engagement, not a player-facing feature.
